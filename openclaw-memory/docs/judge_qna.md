# OpenClaw Memory Engine — 评委 Q&A

## Q1: 你和 ChatGPT 的聊天记录有什么区别？

**评委原话：** "你这个东西和 ChatGPT 的聊天记录/上下文记忆有什么区别？不就是把聊天内容存下来再搜一下吗？"

**考察点：** 是否理解 Memory ≠ Chat History

**最佳回答：** ChatGPT 的聊天记录是线性文本回放，你没法问它"当前项目进展是什么"。我们的 Memory Engine 提取结构化状态——谁负责什么、决定过什么、被什么阻塞了。而且每条记忆都有版本号（负责人从张三换成李四，v1→v2）、证据锚点（来自哪条消息）、有效时间范围（valid_from/valid_to）。你可以查询"上周三开会时的决策状态"（as_of query）。它不是搜索，是协作状态的回放。

## Q2: 你和普通 RAG 有什么区别？

**评委原话：** "现在很多项目用 RAG，把文档切片用向量搜，你这个和 RAG 有什么不同？"

**考察点：** 是否理解结构化记忆 vs 语义搜索

**最佳回答：** RAG 做的是模糊搜索——你搜"谁负责API"出来的是包含 API 相关文本的片段，可能有张三也可能有李四。我们的提取结果是精确的：owner=张三。而且 RAG 没有版本概念——如果李四接替张三，RAG 搜出来的两条记录互相矛盾。我们的引擎会 supersede 旧记录并保留在 history 中。另外，我们不做向量嵌入和相似度计算，而是用三层去重（Identity Key → Content Hash → Semantic Similarity）做精确合并。

## Q3: LLM 提取错了怎么办？

**评委原话：** "如果 LLM 把张三负责的任务说成李四负责了，你们怎么处理？"

**考察点：** 容错设计

**最佳回答：** 三层兜底。第一层是 Schema 校验——非法 JSON、缺失字段会触发回退到规则提取。第二层是证据锚点验证——每个 candidate 的 message_id 必须在输入消息中，伪造的记忆写不进去。第三层是版本管理——错误的记忆写进去后会有一个正确版本 supersede 它，旧版本保留在 history 中不会丢。另外 V1.6 增加了低置信度过滤——ambiguous + confidence≤0.3 的候选在代码层就被丢弃了。

## Q4: 为什么现在主要处理群聊消息？

**评委原话：** "飞书有群聊、文档、任务、会议，你们只做了群聊？其他的什么时候做？"

**考察点：** 优先级判断和对飞书生态的理解深度

**最佳回答：** 群聊是最高频、信息密度最高的协作场景——项目讨论、决策、任务分配、阻塞同步大多发生在群聊中。我们的 `SourceRef.type` 已经在 schema 层面支持了 `"message"`/`"doc"`/`"meeting"`/`"task"` 四种类型，`LarkCliAdapter` 已经封装了 `fetch_doc`/`search_tasks`/`messages_mget` 方法。文档/任务/会议的数据源接入已规划在 P1。

## Q5: 规则提取能覆盖多少办公场景？

**评委原话：** "你的提取器目前基于关键词规则——含'决定'就提取成决策、含'阻塞'就提取成阻塞。这能覆盖多少真实讨论？"

**考察点：** 规则引擎局限性的诚实认知和量化能力

**最佳回答：** 当前基于 RuleBasedExtractor 的 golden set 评测显示 29/30 场景通过（96.7%通过率），8 种闲聊场景零误提取。但这只覆盖含明确关键词的消息。隐含语义（比如"张三在弄前端"表示负责人、"API 还没好"表示阻塞）需要 LLM 才能准确识别。RuleBasedExtractor 的定位是 LLM 的 fallback——当 LLM 返回非法或模糊结果时回退到关键词规则，保证底线准确率。

## Q6: Debounce 为什么不是生产级的？

**评委原话：** "你说 debounce 是安全的 coalescing 不是真正 trailing-edge，这到底什么意思？会不会丢消息？"

**考察点：** 生产化思考深度

**最佳回答：** 当前实现保证：被 debounce 跳过的事件不会被标记为 processed，所以下次非 debounce 调用时一定能处理。但是，如果窗口内来了 10 条消息然后没人再触发调用，这 10 条会一直积压。生产级实现需要像 agent-memory-server 那样用  Redis TTL + 后台调度器做真正的 trailing-edge debounce。我们在 P0 阶段优先确保正确性（不丢消息、不重复标记），P1 会做可靠调度。

## Q7: 去重和 supersede 的边界是什么？

**评委原话：** "你的三层去重逻辑——identity key、hash、semantic merge——什么时候该合并，什么时候该 supersede？如果两条消息说'DDL 从周五改到周三'和'DDL 从周三改到周五'，内容高度相似但语义相反，你怎么处理？"

**考察点：** 对去重逻辑深层边界的理解

**最佳回答：** 第一，identity key 不同时（比如两个不同的 decision key）不会自动关联，它们独立存在。第二，V1.6 做了三层保护：否定极性检测（"不负责"不合并）、owner/status 字段保护（关键字段不同不合并）、时间解析比较。对于 DDL 变更的例子，我们的字符 bigram 相似度会判为高相似（共享大量"DDL从"、"改到"、"周五"等字符），但 V1.6 的字段保护检测到文本语义差异会进 supersede 而非 merge。不过最终，这类语义级别的判断需要真实 LLM 才能做到可靠。

## Q8: 是否复制了开源项目的代码？

**评委原话：** "你们的 opensourse_code 目录里有 mem0、graphiti 等项目的完整代码仓库，extracted_patterns 里有详细的设计笔记——你们的代码是自己写的还是从开源项目'借鉴'的？"

**考察点：** 学术诚信、原创性

**最佳回答：** 所有代码均为原创。`opensourse_code/` 目录下的开源代码和设计笔记是我们做竞品分析的研究记录。最终实现与任何开源项目的代码没有重叠（欢迎评委对比 `src/` 和 `opensourse_code/repos/` 下的文件）。借鉴的是设计思路，比如 agent-memory-server 的 debounce 模式、mem0 的 ADD-only 策略，但这些在我们的代码中都是用新的架构和飞书特定逻辑重新实现的。AGENTS.md 和执行约束文件是明确的原创性证明。

## Q9: 没有完整权限系统，如何保证安全？

**评委原话：** "你说不同群的记忆通过 project_id 隔离——那如果一个人改了 project_id 参数不就访问了其他群的记忆了吗？"

**考察点：** 安全边界的诚实认知

**最佳回答：** 当前基于 project_id 的隔离是软隔离，确实没有真实身份认证。但在 P0 场景中，所有操作都在本地运行，不存在网络攻击面。我们的安全设计重点在"防止引擎自动执行危险操作"——SafetyPolicy 拦截所有写入命令，`docs +create --dry-run` 被明确禁止。生产环境的权限隔离需要飞书 OAuth scope 集成，已规划在 P2。

## Q10: 30% 的代码是测试，但有多少真正验证了引擎的有效性？

**评委原话：** "你们有 59 个测试，我看了很多是测单个函数的——author_map 能不能正确提取、time_ref 能不能正确排序。但有多少测试验证了'这个引擎在真实办公场景中有效'？"

**考察点：** 测试体系的有效性 vs 数量

**最佳回答：** 这是一个很好的观察。V1.6 之前我们的测试以单元测试为主。V1.6 做了三件事来弥补：第一，增加了端到端集成测试（4 条飞书消息输入→engine→检查最终 memory state）。第二，构建了 30 条标注样本的 golden set，覆盖 12 种办公场景，可以定量评估 precision/recall。第三，增加了 bi-temporal 的 as_of 查询测试和旧数据兼容性测试。但更完整的评测确实需要接入真实 LLM 并对比 LLM vs RuleBased 的效果——这是 P1 的优先事项。
