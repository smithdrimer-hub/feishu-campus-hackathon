## 第二部分：Memory Engine 操作

展示系统如何拉取信息、提取记忆、更新状态。

### 2.1 基础操作：同步 + 查看状态（安全模式）

```bash
cd openclaw-memory

# 同步群聊消息，生成状态面板（dry-run 不发消息）
python scripts/demo_e2e_pipeline.py \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --project-id demo \
  --data-dir data/demo \
  --dry-run
```

**输出**：终端显示提取到的记忆数量、类型分布、状态面板内容。

**演示要点**：系统从群聊中自动识别出了目标、负责人、决策、阻塞、下一步、DDL、暂缓。这不是关键词匹配——"倾向于用 React 18"被标记为 preference（待审核），"确定采用微服务"被标记为 confirmed（正式决策）。

### 2.2 全数据源同步

```bash
# 拉取群聊 + 文档 + 任务 + 日历 + 会议纪要
python scripts/demo_e2e_pipeline.py \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --project-id demo \
  --data-dir data/demo \
  --doc-id Vl8odny0EopGeXxaDuFcbEgfnVd \
  --task-query "Memory" \
  --sync-calendar \
  --sync-minutes \
  --dry-run
```

**演示要点**：5 个数据源的信息被整合到一个项目视图里。文档里的分工、任务里的状态、日历里的会议结论，全部进入记忆。

### 2.3 发送状态面板到群

```bash
# 去掉 --dry-run，面板会发送到飞书群
python scripts/demo_e2e_pipeline.py \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --project-id demo \
  --data-dir data/demo
```

**演示要点**：飞书群里会收到一条格式化的状态面板消息。这是闭环的第一步——从飞书来，回到飞书去。

### 2.4 启动触发引擎（自动执行）

```bash
# 触发引擎自动检测新记忆 → 创建任务、发送提醒
python scripts/demo_e2e_pipeline.py \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --project-id demo-trigger \
  --data-dir data/demo_trigger \
  --trigger --mode auto
```

**演示要点**：系统不只是记住信息——它会主动行动。检测到新阻塞 → 群里发提醒。检测到 deadline 临近 + 有阻塞 → 发送风险预警。检测到下一步+负责人 → 创建飞书任务。

### 2.5 审核台操作

```bash
# 列出待审核记忆
python scripts/demo_review_desk.py \
  --data-dir data/demo \
  --project-id demo \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4

# 批准一条记忆
python scripts/demo_review_desk.py \
  --data-dir data/demo \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --approve mem_xxx

# 驳回一条记忆
python scripts/demo_review_desk.py \
  --data-dir data/demo \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --reject mem_xxx

# 合并两条相似记忆
python scripts/demo_review_desk.py \
  --data-dir data/demo \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --merge mem_aaa --from mem_bbb

# 更新阻塞状态
python scripts/demo_review_desk.py \
  --data-dir data/demo \
  --chat-id oc_e1c6a2c2a42b67606b91ad69bab226f4 \
  --approve mem_xxx --blocker-status resolved
```

**演示要点**：系统做候选识别，管家做最终裁定。每一项待审核记忆都有完整原文证据和可点击的飞书链接。管家验证通过飞书群成员身份——必须是群成员才能操作。

### 2.6 站会摘要 + 确认清单


## 第三部分：可演示的功能清单与展示方法

### 3.1 核心差异化能力（必演示）

| 功能 | 演示方法 | 评委看到的效果 |
|------|---------|---------------|
| **8 种状态提取** | 发送 10 条协作消息 → dry-run 查看提取结果 | 系统从群聊中自动识别目标/负责人/决策/阻塞/DDL/下一步/暂缓/请假 |
| **决策分层** | 发送"确定用X"和"倾向于Y"→ 审核台查看强度标记 | non-confirmed 的决策不会自动生效，进入审核台 |
| **冲突检测** | 发送"用React"+"改用Remix"→ 审核台查看冲突标记 | 系统发现矛盾，不强行覆盖，交给人判断 |
| **审核台** | 展示待审核列表 + approve/reject 操作 | 每条记忆有原文证据+飞书链接，管家逐条确认 |
| **触发闭环** | 发送"阻塞：XXX"→ 群内收到预警消息 | 从群聊来 → 自动分析 → 回到群聊提醒 |

### 3.2 增强能力（建议演示）

| 功能 | 演示方法 |
|------|---------|
| **阻塞生命周期** | 审核台标记 blocker 为 resolved → 面板显示在"最近已解决" |
| **多负责人共存** | 发送"张三负责前端，李四负责后端"→ 面板显示两个 owner |
| **低置信度提问** | 发送模糊的 next_step → preview 模式查看生成的确认消息 |
| **站会摘要** | `--standup` 参数 → 三段式站会格式 |
| **WebSocket 实时** | `--ws-check` 运行 → 发消息 → 秒级收到事件 |
| **每日早报** | `--once` 运行 → 查看生成的早报内容 |
| **多数据源** | `--doc-id` + `--task-query` + `--sync-calendar` 一起跑 |

### 3.3 推荐演示流程（10 分钟版）

| 时间 | 环节 | 操作 |
|------|------|------|
| 0:00 | 开场：问题 | 展示飞书群里散落的 10 条消息——"如果你要接手这个项目，多久能搞清楚？" |
| 1:00 | 感知：提取 | 运行 `--dry-run`，展示系统提取了 8 种状态 |
| 2:00 | 亮点：决策分层 | 展示"倾向于 React 18"被标记为 preference（待审核），"确定微服务"标记为 confirmed |
| 3:00 | 亮点：冲突检测 | 展示"用React"和"用Remix"被检测为冲突 |
| 4:00 | 治理：审核台 | 展示 CLI 审核界面——证据原文、飞书链接、approve/reject |
| 5:00 | 执行：触发引擎 | 展示系统自动创建任务、发送预警 |
| 6:00 | 闭环：状态面板 | 发送面板到群——"从飞书来，回到飞书去" |
| 7:00 | 增强：阻塞生命周期 | 标记 resolved → 面板更新 |
| 8:00 | 增强：站会+早报 | 展示站会摘要和每日早报 |
| 9:00 | 实时：WebSocket | `--ws-check` 运行中，发一条消息，3 秒内收到事件 |
| 10:00 | 总结 | "AI 做候选识别，管家做最终裁定。完整闭环。" |
