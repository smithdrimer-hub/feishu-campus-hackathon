# OpenClaw Memory Engine — 决赛准备状态报告

> 评估日期：2026-05-07 | 距决赛约 5 天

## 一、按评分维度自评

### 维度 1：完整性与价值（40 分）→ 自评 32/40

**解决什么痛点**：
企业协作中，沟通发生在飞书群里，但状态散落在不同时间、不同人的消息中。有人请假或离职时，接手的人需要翻几十页聊天记录。系统自动提取 8 种协作状态 + 6 种二阶模式，绑定原始证据，生成交接摘要和状态面板。

**AI 的关键作用**：
不是关键词匹配。DeepSeek V4 Pro 在 Selector 模式下处理模糊语义（"那就先这样吧""在弄了""还没好"），规则处理精确信号（"负责人：张三""DDL 周五"）。Hybrid 8 条件触发体系确保 LLM 只在需要时调用。

**闭环程度**：已完成，但有薄弱点：

| 环节 | 状态 | 备注 |
|------|------|------|
| 飞书→提取 | ✅ | 5 个数据源（群聊/文档/任务/日历/会议纪要） |
| 提取→审核 | ✅ | decision_strength + review_status + 审核台 CLI |
| 审核→执行 | ✅ | 5 条触发规则 + 冷却 + 安全策略 |
| 执行→飞书 | ✅ | 创建任务/发送提醒/风险预警 → 飞书群 |
| 飞书→确认 | ⚠️ | R4 闭环已实现，但未被持续验证 |
| 执行→记忆回写 | ⚠️ | task_guid 记录但未自动回流 |

**Demo 稳定性**：327 测试全过、Golden Set 150/150、6 次真实飞书端到端验证。Windows 下卡片自动降级 markdown 兜底。

**实际价值**：10 人团队每月节省 8-12 小时信息检索时间。交接时间从 1-2 小时降到 5-10 分钟。

### 维度 2：创新性（20 分）→ 自评 16/20

**AI 相关创新点**：

1. **Selector 模式**（自研设计）：规则的职责从"提取"变为"判断是否有把握"。精确信号→规则，模糊信号→LLM，纯问题→跳过。开源代码中没有对应方案。

2. **Hybrid 8 条件 LLM 触发体系**：不是简单的"规则不够就调 LLM"。8 个条件分层检测，条件(h)自研——规则产出存在内部冲突时调 LLM 二次判断。

3. **Work Pattern Memory**：已有 MemoryItem 的二阶归纳层。6 种模式，不扫原始消息。不是"员工画像"，而是"可审计的协作事实"。

4. **决策分层 + 冲突检测**：4 级强度 + 同主题矛盾并排展示 + 管家裁定。

**差异化亮点**：
- 对比 mem0/graphiti/langmem：飞书原生深度（6 个模块统一接入）、证据链可追溯（sender+URL+excerpt）、安全策略（只读自动/写入确认/dry-run 零副作用）
- 对比传统 RAG：结构化状态机 vs 向量片段。可审计 vs 黑盒

**可复用性**：adapter 模式留 backend 可替换接口（飞书→Slack/Teams 只需换 adapter）。LLM provider 接口可插拔（DeepSeek/OpenAI/Claude 都能接）。

### 维度 3：AI 技术实现（20 分）→ 自评 17/20

**AI 技术深度**：
- LLM → Schema 校验 → 规则兜底 → excerpt 原文验证（4 层可信链路）
- JSON mode + temperature=0 确定性提取
- 4 层语义去重（identity key / content hash / bigram / 跨 key 决策覆盖）
- Confidence-based routing（高置信→自动 / 中置信→审核台 / 低置信→主动提问）
- Selector 决策矩阵（精确/模糊/纯问题/极简 四级分流）
- ChromaDB + text-embedding-3-small 向量语义搜索

**架构合理性**：
- adapter / extractor / engine / store / trigger / pattern 6 层正交分离
- 类/函数职责单一，依赖方向自上而下

**工程规范**：
- 327 单元测试、150 Golden Set
- dataclass schema、YAML 配置可外置
- 安全策略：只读自动/写入确认/dry-run 零副作用（含本地磁盘）
- subprocess 超时 + 原子写入 + 损坏恢复 + 孤儿进程清理

**可扩展性**：
- MemoryStore 支持 limit/offset 分页
- 倒排索引 O(1) 查询
- 向量搜索可插拔（Fake/OpenAI）
- WebSocket 事件监听支持心跳/重连/事件路由
- lark-cli adapter 模式留 backend 可替换接口

## 三、全链路审计（提取→存储→治理→执行→呈现→反馈）

### 数据摄入

| 现状 | 问题 | 比赛影响 |
|------|------|---------|
| 5 数据源通路已验证 | 无增量跟踪，每次全量 | 演示数据量小，无影响 |
| 文档 hash 检测更新 | hash 路径可能冲突 | 低风险 |
| auto_runner 循环 | 子进程可能残留 | Ctrl+C 后可手动清理 |

### 提取

| 现状 | 问题 | 比赛影响 |
|------|------|---------|
| Selector 模式在 Hybrid 中工作 | 仅 Hybrid 可用，RuleOnly 仍全量提取 | 演示需传 `--hybrid` 才展示 Selector |
| `_safe_llm_extract` 失败回退 | API 挂掉时静默降级为规则 | 评委看不出 LLM 在工作 |
| 无质量反馈 | 驳回的记忆不会反向优化提取 | 长期问题，比赛不涉及 |

### 存储与去重

| 现状 | 问题 | 比赛影响 |
|------|------|---------|
| 4 层去重 | Layer 4 >200 条跳过 | 演示数据量小不触发 |
| bi-temporal | `valid_to` 仅 supersede 时设置 | 陈旧记忆不自动归档 |
| 全量读写 | 93 条 0.1s | 比赛不涉及 |

### 治理

| 现状 | 问题 | 比赛影响 |
|------|------|---------|
| 审核台 CLI 完整 | 无批量操作 | 演示可逐个展示 |
| 冲突检测仅 cover decision | owner/blocker 不一致不检测 | 评委问"文档和任务冲突呢"无法展示 |
| 驳回可被重提取 | 无永久忽略列表 | 同样内容反复进审核 |

### 执行

| 现状 | 问题 | 比赛影响 |
|------|------|---------|
| 5 条触发规则 | R4 回复解析是关键词匹配 | "都行/随便"无法处理 |
| 冷却进程内缓存 | 重启丢失 | 演示单次运行不涉及 |

### 呈现

| 现状 | 问题 | 比赛影响 |
|------|------|---------|
| 交互式卡片 | Windows 中文卡片降级 markdown | Mac 演示效果更好 |
| 状态面板 | 无时间分组 | 信息堆砌而非"最近变化" |
| 每日早报 | "昨日变化"依赖 updated_at | 同批同步时间戳相同 |

### 反馈闭环

| 现状 | 问题 | 比赛影响 |
|------|------|---------|
| R4 回复解析 | "确认1,2"精确匹配 | "都行/无所谓"无法处理 |
| 审核无统计 | 不记录通过率/驳回率 | 评委问效果无法量化 |
| Pattern 已持久化 | 未接入审核台 | 管家看不到协作模式 |
