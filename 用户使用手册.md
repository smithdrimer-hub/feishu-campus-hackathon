# OpenClaw Memory Engine — 产品说明


## 这个系统是什么

OpenClaw Memory Engine 是一个嵌入飞书办公生态的**团队协作记忆引擎**。它从飞书群聊、文档、任务、日历、会议纪要中自动识别团队的协作状态——谁在负责什么、做过什么决策、被什么卡住了、下一步该做什么——然后生成清晰的项目状态面板，并在合适时机主动创建任务和发送提醒。

**和聊天记录搜索的根本区别**：聊天记录搜索是"你问它找"——你必须知道自己在找什么。记忆引擎是"它替你整理"——你打开就能看到当前项目的完整状态，包括不同时间、不同人说的事情之间的关联。比如系统会发现"周五有 deadline"和"还有3个阻塞没解决"这两件事之间存在风险关系，然后主动预警。


## 一、日常办公中它能做什么

### 场景 1：有人请假或离职，接手的人需要快速了解项目状态

**以前的做法**：翻几十页群聊记录、打开多个文档、问好几个人。通常需要 1-2 小时才能拼凑出"现在是什么情况"。

**系统怎么做**：
- 自动生成一份**交接摘要**，用 8 个维度讲清楚项目当前状态：目标是什么、谁在负责什么、做过哪些关键决策、有什么截止日期、哪些事被暂缓了、当前被什么阻塞、下一步要做什么、成员的可用状态
- 每一条结论后面都有**证据链接**，点击直接跳转到原始飞书消息——不是系统"编"的，可以看到原始出处
- 历史版本可追溯——"这个决策后来被推翻过吗？"可以查到完整变更链

**当前能做到的程度**：交接摘要功能完整，8 个维度覆盖了中断续办所需的全部信息。证据链可以追溯到原始消息。但摘要是手动生成的（运行命令），不会在有人离职时自动触发。

### 场景 2：每天站会，快速同步进展

**以前的做法**：10 个人 × 每人回忆 3-5 分钟 = 30-50 分钟的站会。

**系统怎么做**：
- 自动生成**项目状态面板**（一个 Markdown 格式的结构化报告），包含当前负责人、已确认的决策、待定的决策、进行中的任务、阻塞与风险、建议的下一步
- 另外提供**站会摘要格式**（昨日进展 / 今日计划 / 阻塞风险三段式），更贴近站会节奏
- 可以一键发送到飞书群

**当前能做到的程度**：状态面板和站会摘要都已可用，能自动生成并发送到飞书群。但"昨日进展"部分依赖 `updated_at` 时间戳——如果消息是同一批同步的，时间戳会集中在同一时刻，区分不出"昨天"和"更早"。

### 场景 3：口头说的事变成正式任务，不会忘

**以前的做法**：群里有人说"李四你跟设计师沟通一下"，过两天大家都忘了，没人跟进。

**系统怎么做**：
- 识别到含有"下一步行动 + 负责人"的消息时，自动在飞书中**创建任务**，分配给对应的人
- 如果系统能在飞书通讯录中找到这个人的 open_id，任务会直接分配；找不到则标记为"需人工确认"
- 同一个任务 24 小时内不会重复创建

**当前能做到的程度**：以 user 身份发消息时，owner 提取基本准确。以 bot 身份发消息时，sender 显示为 bot 名称而非真实姓名，导致 owner 提取为空——这是飞书权限限制，不是系统 bug。任务创建、文档创建均已通过真实飞书 API 验证。

### 场景 4：被某件事卡住了，依赖方及时知道

**以前的做法**：A 等 B 的设计稿，B 不知道 A 在等。等到站会发现已经卡了 3 天。

**系统怎么做**：
- 检测到新出现的阻塞（如"等待设计稿，前端动不了"），自动在群里**发送提醒消息**
- 同一轮如果发现多个阻塞，会**合并成一条消息**，避免刷屏
- 如果同时有临近的 deadline（3 天内）+ 未解决的阻塞，系统会发**风险预警**——这是跨记忆推理，不是简单的关键词匹配

**当前能做到的程度**：阻塞提醒和风险预警均已通过真实飞书群验证。阻塞有 5 种生命周期状态（open / acknowledged / waiting_external / resolved / obsolete），已解决的阻塞会移到"最近已解决"区，7 天后自动归档。但阻塞状态的变更是手动的（需要管家在审核台操作），系统不会自动检测"阻塞已解决"。

### 场景 5：开会做了决定，会后自动整理

**以前的做法**：开会讨论了什么、做了什么决策，散会就忘。翻会议纪要又太长。

**系统怎么做**：
- 从飞书妙记（会议纪要）中读取 AI 自动生成的总结和待办项
- 将这些信息整合到项目状态中，和其他来源（群聊、文档、任务）的信息统一管理
- 可以生成**确认清单**——"系统识别到以下决策和待办，请确认是否正确"——发送到群里让参会人确认

**当前能做到的程度**：会议纪要的 API 通路已验证（真实妙记数据可读取）。确认清单的生成逻辑已完成。但会议纪要依赖飞书 AI 的总结质量——如果妙记的 AI 总结不准确，系统提取的结果也会受影响。

### 场景 6：多个来源的信息互相矛盾时，系统能发现

**以前的做法**：群聊里说用 React，文档里写用 Vue，没人注意到这个矛盾，直到开发时才发现。

**系统怎么做**：
- 当两个不同来源的决策描述的是同一件事但结论不同时，系统会标记为**冲突**，而不是强行覆盖
- 冲突项进入审核台，管家可以看到双方证据并排展示，选择保留哪一个
- 如果消息中有明确的覆盖信号（"改为""不再用"），系统会正常执行覆盖；只有在没有明确覆盖信号、但内容明显矛盾时，才标记冲突

**当前能做到的程度**：冲突检测功能已完成。在真实测试中成功检测到"倾向于 React 18"和"改用 Remix"之间的冲突。但冲突检测仅覆盖 decision 类型，不覆盖 owner、blocker 等其他类型的冲突。

### 场景 7：项目管家审核存疑的记忆

**以前的做法**：AI 提取的内容直接生效，不知道哪些可靠、哪些存疑。

**系统怎么做**：
- 自动为每条记忆标记审核状态：正常的自动通过（auto_approved），存疑的进入待审核（needs_review）
- 以下情况会自动标记为需要审核：决策是非确认级别的（tentative/preference/discussion）、记忆覆盖了旧版本、置信度低于 0.6、没有证据来源、存在冲突
- 管家通过审核台逐条确认、驳回或修改——驳回的记忆移到历史，修改后的记忆正常生效
- 审核台会展示每条记忆的**完整原文证据**和**可点击的飞书链接**，方便管家快速判断
- 管家身份通过飞书群成员验证——必须是群成员才能操作审核台

**当前能做到的程度**：审核台功能完整，已在真实测试中验证。但只有命令行界面，没有飞书交互界面——评委看到的是终端输出。管家身份验证使用的是飞书群成员 API（`chat.members get`），而不是飞书 OAuth。

### 场景 8：管理者一眼看到所有项目的风险

**以前的做法**：靠 PM 逐个问、逐个检查，信息滞后 1-3 天。

**系统怎么做**：
- 支持**跨项目视图**——一个人如果在多个项目中有任务，可以汇总显示
- 阻塞数量、严重程度一目了然
- 触发引擎会根据规则自动发送风险预警到群

**当前能做到的程度**：跨项目视图功能完整，但 owner 匹配是基于姓名字符串的——如果同一个人的名字在不同项目中有不同写法（"张三"/"小张"/"张工"），无法自动识别为同一人。


## 二、它能从哪些地方获取信息

系统从飞书的 **5 个模块**中拉取信息。每个模块的接入程度不同：

| 信息来源 | 接入程度 | 说明 |
|---------|---------|------|
| **群聊消息** | ★★★★★ 主力验证 | 自动拉取，支持分页。以 user 身份发送的消息可以正确提取发送者姓名 |
| **飞书文档** | ★★★ 可用，需 stdin 创建 | 可读取文档内容并按章节拆分提取。文档需通过 stdin 方式创建才能保证内容完整 |
| **飞书任务** | ★★★ 可用 | 可搜索并同步任务状态。创建任务功能已通过真实 API 验证 |
| **日历日程** | ★★★ 可用 | 可同步本周日程。日程中的描述文本会被提取 |
| **会议纪要(妙记)** | ★★★ 可用 | 可读取飞书 AI 生成的会议总结和待办。实际测试中成功读取 2 条真实纪要 |
| **审批实例** | ★★ 通路就绪 | API 通路验证，但测试账号无待审批数据，缺乏真实端到端验证 |

> **诚实说明**：不是"6 个模块无缝统一"。群聊是唯一经过反复测试的场景。文档、任务、日历、会议纪要的 API 通道已验证可以拉取数据，但缺乏大量真实数据的稳定性测试。审批模块的端到端验证受限于测试账号的数据。


## 三、它能识别哪些协作信息（8 种状态类型）

系统从消息中自动识别 8 种协作状态。识别方式有两种：**规则匹配**（关键词+正则，快速且确定性强）和 **LLM 语义理解**（覆盖规则无法处理的隐式表达，但速度较慢，约 65 秒/50 条消息）。

### 每种状态的识别能力和边界

#### 项目目标（project_goal）
- **能识别的**："目标：完成用户中心重构""这周集中把 XX 搞完""goal: launch MVP"
- **会误判的**：含"目标"但不是项目目标的句子（如"目标用户是年轻人"）
- **会遗漏的**：不含"目标"关键词的目标表达（如"重点是 XX""当前主要工作是 XX"）
- **实现方式**：关键词匹配

#### 负责人（owner）
- **能识别的**："负责人：张三""由李四负责 API""张三负责前端，李四负责后端""分工：我负责XX"
- **会误判的**："由系统处理"（把"系统"当人名）、"由此处理"（把"此"当人名）。第一人称"我"已排除
- **会遗漏的**："我来做前端"（第一人称+无"负责"关键词）、"让张三处理一下"（无标准负责人格式）
- **实现方式**：6 种正则模式 + domain 提取
- **重要**：多人可以共存（V1.15 修复了之前的覆盖 bug）。以 user 身份发消息时 owner 提取准确，以 bot 身份发消息时 sender 显示为 bot 名称

#### 决策（decision）
- **能识别的**："确定用 React""决定迁移到云""采用微服务""不再用单体"
- **做了分层的**：系统将决策分为 4 级——
  - **discussion**（讨论中）："考虑用微服务""要不要迁移"
  - **preference**（偏好表达）："倾向于 React""建议用 Go"
  - **tentative**（暂定方向）："那就先这样做""暂时按这个方案"
  - **confirmed**（正式确认）："确定用 React""最终方案是 XX""就这么定了"
- **只有 confirmed 级别的决策会直接进入正式状态面板**。其余级别默认进入待审核（needs_review），需要管家确认
- **会误判的**："我不确定这个方案"（含"确定"但语义相反，V1.15 Hybrid 条件(h)已覆盖此场景，LLM 模式下会二次判断）
- **会遗漏的**："那就先这样吧""算了还是用 X"（无标准决策关键词，LLM 模式可补充识别）
- **实现方式**：8 个关键词 + 强度推断；LLM 模式下 Prompt 辅助判断

#### 截止时间（deadline）
- **能识别的**："DDL 到下周五""截止日期改到周三""延期到明天"
- **会误判的**："改到另一个方案"（含"改到"但非截止日期语境）
- **会遗漏的**："周五交""周三前完成"（不含"DDL/截止/deadline"关键词）
- **日期解析**：支持 8 种中文相对时间表达（明天/后天/周五/下周三/5月10日/3天后/下周/下个月）
- **实现方式**：关键词匹配 + 日期裁剪；日期解析器将中文时间转为具体日期

#### 阻塞与风险（blocker）
- **能识别的**："阻塞：等待设计稿""依赖第三方 API""风险：服务器资源不足"
- **生命周期**：open → acknowledged → waiting_external → resolved → obsolete；已解决的阻塞 7 天后自动归档
- **会误判的**："风险是存在的但可控"（含"风险"但表达的是风险可控）。V1.15 增加了转折词检测（"但/不过"），LLM 模式会二次判断
- **会遗漏的**："还没好""动不了""来不及""卡在审批"（无"阻塞/风险/依赖"等关键词，LLM 模式可补充）
- **实现方式**：6 个关键词 + 解除信号过滤；5 种生命周期状态

#### 下一步行动（next_step）—— 精度最低的类型
- **能识别的**："下一步：完成 API 设计""李四需要跟设计师沟通""请张三更新文档""待办：测试"
- **会误判的**："请问这个怎么配"（含"请"但只是技术咨询）、"需要确认一下"（含"需要"但不构成具体行动）。**这是误判率最高的类型，约每 3 条提取就有 1 条不是真正的下一步行动**
- **会遗漏的**："记得提交代码""别忘了更新文档"（无"下一步/需要/请"关键词）
- **实现方式**：6 个关键词 + 动作主语提取。V1.15 新增了"李四需要做 X"模式识别，可从中提取 owner

#### 暂缓事项（deferred）
- **能识别的**："暂缓：国际化""先不做移动端""延期到下个版本"
- **会误判的**："先不急"（在日常对话中频率高，但不一定有协作含义）
- **会遗漏的**："放一放""缓一缓""hold 住"（不在关键词列表中）
- **实现方式**：5 个关键词

#### 成员状态（member_status）
- **能识别的**："我这周请假""明天出差""习惯用 Figma""擅长后端架构"
- **会误判的**："不在状态"（含"不在"但非请假语境）
- **会遗漏的**："调休""年假""去上海了""全天会议"（不在关键词列表中）
- **实现方式**：6 个关键词 + 邻近匹配（"请个假"也能匹配"请假"）+ value 裁剪

### 关于准确率的诚实说明

在 150 条人工标注的评测集上，规则模式的表现（按状态类型分解）：

| 类型 | 精确率 | 召回率 | F1 | 一句话 |
|------|--------|--------|-----|--------|
| blocker | 1.00 | 1.00 | 1.00 | 关键词精准，误判低 |
| deadline | 1.00 | 1.00 | 1.00 | 同上 |
| decision | 1.00 | 1.00 | 1.00 | 评测集上的决策信号明确，真实办公会有边界 |
| owner | 1.00 | 1.00 | 1.00 | 格式化的负责人表达覆盖完整 |
| project_goal | 1.00 | 1.00 | 1.00 | "目标"关键词精准但会漏隐式表达 |
| member_status | 1.00 | 1.00 | 1.00 | 评测集覆盖了标准请假/出差/习惯表达 |
| deferred | 0.80 | 1.00 | 0.89 | 偶有将非暂缓标记为暂缓 |
| next_step | **0.65** | 1.00 | 0.79 | "需要""请"太宽泛，约 1/3 为误判 |

**正确解读**：
- 以上是**150 条人工构造评测集上的规则匹配表现**，不是真实群聊的语义理解准确率。评测集的文本是精心构造的、含有明确信号词的——真实群聊有错别字、半截话、表情包、多人插话，准确率会低于评测集
- Recall 全 1.00 是因为评测集覆盖了所有关键词场景——真实群聊中不含关键词的隐式表达会被遗漏
- next_step 的 0.65 精确率是系统设计中**故意偏向保守**的代价——宁可用宽泛的关键词多提取一些，也不遗漏真正的行动项。误判的项置信度低（0.62），可以通过审核台过滤
- **LLM（Hybrid）模式可以补充识别规则遗漏的隐式表达**，但对规则已经提取但提取错了的情况，只在特定条件下才会调用 LLM 二次判断（V1.15 新增的 4 种冲突检测条件）


## 四、它会主动做什么（触发引擎）

系统有 3 条自动触发规则，在提取完成后按需执行。所有执行操作都有冷却机制（24 小时内不重复）。

| 规则 | 触发条件 | 动作 | 当前可靠性 |
|------|---------|------|-----------|
| R1: 新任务→飞书任务 | 新提取到 next_step + owner 不为空 | 在飞书创建任务，分配给 owner | 中等——owner 提取受 bot/user 身份影响 |
| R2: 新阻塞→群提醒 | 新提取到 blocker | 发送提醒到群（多个阻塞合并） | 较高——阻塞关键词精准 |
| R3: 截止+阻塞→预警 | deadline ≤3天 + 存在未解决阻塞 | 发送风险预警（跨记忆推理） | 较高——但日期解析仅覆盖 8 种模式 |

**执行模式**：
- `preview`：只展示提案，不执行（默认最安全）
- `auto`：自动执行所有提案
- 所有写入操作有飞书安全策略保护

**当前不触发的场景**：
- owner 变更（"张三换成了李四"）不会通知
- 阻塞已解决不会自动通知依赖方（需要手动在审核台标记 resolved）
- 长期无更新的任务不会自动提醒（没有定时扫描）


## 五、你能看到什么输出

### 1. 项目状态面板
每次同步消息后自动生成，可一键发送到飞书群。包含：负责人、已确认决策、待定决策、进行中任务、阻塞与风险、建议下一步、最近已解决的阻塞。

### 2. 交接摘要
8 个维度（目标/负责人/决策/截止/暂缓/阻塞/下一步/成员状态），每条有证据链接和置信度。

### 3. 站会摘要
昨日进展 / 今日计划 / 阻塞风险三段式。

### 4. 审核台（命令行）
列出待审核记忆，逐条展示完整原文和飞书链接。管家可以确认、驳回、修改、合并。

### 5. 确认清单
会议纪要后生成，列出系统识别到的决策、待办和风险，让参会人确认。


## 六、重要：当前功能边界与局限

以下逐项说明每个功能的**当前实际状态**，帮助你在演示时准确表述系统能做到什么、不能做到什么。

### 提取层面

| 能做到 | 不能做到 |
|--------|---------|
| 识别含明确关键词的协作表达（如"负责人：""阻塞：""DDL"） | 识别完全不含关键词的隐式表达（LLM 模式可部分补充，但非全覆盖） |
| 从群聊消息中提取 8 种协作状态 | 从图片、语音、文件、表情包中提取信息 |
| 支持中英文混合消息 | 处理方言、网络黑话、拼音缩写 |
| user 身份消息正确提取 sender 姓名 | bot 身份消息提取真实姓名（受飞书权限限制） |
| 多人负责人共存（V1.15 修复） | 区分 owner/co-owner/reviewer/backup 等角色 |

### 决策层面

| 能做到 | 不能做到 |
|--------|---------|
| 区分 4 级决策强度 | 100% 准确区分——"我考虑一下" vs "考虑用微服务" 规则无法区分 |
| confirmed 级决策自动进入正式面板 | 自动判断什么是"正式确认"——依赖关键词推断 |
| 检测两个决策之间的冲突 | 检测所有类型的冲突——当前仅覆盖 decision 类型 |
| LLM 模式下对模糊决策二次判断 | 规则模式下对假阳性决策自动纠正 |

### 执行层面

| 能做到 | 不能做到 |
|--------|---------|
| 新阻塞自动发群提醒 | @具体的人（bot 身份消息没有真实姓名映射） |
| 自动创建飞书任务 | 自动设置任务截止日期（未从消息中提取时间信息填入） |
| 24 小时冷却防重复 | 跨会话冷却（重启后内存缓存丢失，仅文件级冷却生效） |

### 审核层面

| 能做到 | 不能做到 |
|--------|---------|
| 自动标记 5 种高风险记忆 | 在飞书界面上审核（当前仅命令行） |
| 展示完整原文证据和飞书链接 | 永久忽略某类误判（驳回后再次提取仍会出现） |
| 管家身份通过飞书群成员验证 | 区分群内不同角色（owner/admin/member）的权限 |
| 合并相似记忆 | 自动识别所有可合并的重复项（需管家手动发现） |

### 数据源层面

| 能做到 | 不能做到 |
|--------|---------|
| 群聊消息自动同步（主力验证） | 实时监听——当前是手动触发同步 |
| 文档/任务/日历/会议纪要接入 | 每个模块都经过充分的不稳定网络和边界数据测试 |
| 从文档章节中提取信息 | 从嵌入的表格、图片、思维导图中提取信息 |
| 飞书妙记的 AI 总结 | 改善妙记本身的质量（依赖飞书 AI 输出） |

### 系统整体

| 能做到 | 不能做到 |
|--------|---------|
| 手动触发同步后全自动处理 | 开机自启、常驻后台、定时自动运行 |
| 单项目状态管理 | 多项目间的依赖关系分析 |
| 命令行操作 | 飞书群内 @机器人 交互 |
| 310 个单元测试全部通过 | 覆盖所有真实办公的边界情况 |
